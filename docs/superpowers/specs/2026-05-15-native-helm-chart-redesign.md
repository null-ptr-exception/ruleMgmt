# Native Helm Chart Redesign

## Goal

Redesign the PrometheusRule template/alert flow to use standard Helm chart structure with zero custom metadata. Template developers build real Helm charts. Alert users fill in values via a tabular UI. Variable definitions use `values.schema.json` (Helm's native schema mechanism).

## Scope

- PrometheusRule template development and alert user flows only
- Alertmanager config, receivers, and gitops-deploy are out of scope (unchanged)

## Storage Format

Each chart is a standard Helm chart on disk:

```
templates/<chart-name>/<version>/
  Chart.yaml              # name, description, version (Helm native)
  values.yaml             # default values
  values.schema.json      # variable definitions as JSON Schema
  templates/
    prometheus-rule.yaml   # PrometheusRule with {{ .Values.xxx }}
    recording-rules.yaml   # optional additional template files
```

No custom metadata files. `helm lint` and `helm template` work out of the box.

### Chart.yaml

Standard Helm chart metadata. Name and description are read from here — not duplicated in values.yaml.

```yaml
apiVersion: v2
name: tablespace-usage
description: Database tablespace utilization alerts with multi-tier severity
version: 1.0.0
type: application
```

### values.yaml

Default values for the alert user. The top-level `instances` array holds one entry per table row.

```yaml
instances:
  - db_name: mydb
    tablespace_name: users
    info: 30
    warn: 50
    critical: 70
    severe: 95
```

### values.schema.json

Defines the columns of the alert user's table. Standard JSON Schema.

```json
{
  "$schema": "https://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "instances": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "db_name": {
            "type": "string",
            "description": "Database name"
          },
          "tablespace_name": {
            "type": "string",
            "description": "Tablespace name"
          },
          "info": {
            "type": "number",
            "description": "Info threshold (%)",
            "default": 30
          },
          "warn": {
            "type": "number",
            "description": "Warning threshold (%)",
            "default": 50
          },
          "critical": {
            "type": "number",
            "description": "Critical threshold (%)",
            "default": 70
          },
          "severe": {
            "type": "number",
            "description": "Severe threshold (%)",
            "default": 95
          }
        },
        "required": ["db_name", "tablespace_name"]
      }
    }
  }
}
```

### templates/prometheus-rule.yaml

Auto-generated by the UI from the fixed fields (name, description, expr) and dynamic vars. The template developer can customize it afterward.

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: {{ .Release.Name }}-tablespace-usage
  labels:
    app.kubernetes.io/managed-by: Helm
spec:
  groups:
    - name: tablespace-usage
      rules:
        {{- range .Values.instances }}
        {{- $inst := . }}
        {{- range $severity, $threshold := dict "info" .info "warn" .warn "critical" .critical "severe" .severe }}
        - alert: tablespace-{{ $severity }}-{{ $inst.db_name }}-{{ $inst.tablespace_name }}
          expr: 'tablespace_usage_percent{db="{{ $inst.db_name }}", tablespace="{{ $inst.tablespace_name }}"} > {{ $threshold }}'
          for: 5m
          labels:
            severity: {{ $severity }}
            db: {{ $inst.db_name }}
            tablespace: {{ $inst.tablespace_name }}
          annotations:
            description: 'Tablespace {{ $inst.tablespace_name }} on {{ $inst.db_name }} exceeds {{ $threshold }}%'
        {{- end }}
        {{- end }}
```

## Template Developer UX

### Editor Layout

Left sidebar: chart list (name/version tree). Main area: editing surface.

### Fixed Fields (form at top)

Every chart has these — the UI renders them as form inputs:

- **Name** — alert name pattern, e.g. `tablespace-{{ $severity }}-{{ $inst.db_name }}`
- **Description** — annotation template, e.g. `Tablespace {{ .tablespace_name }} exceeds {{ $threshold }}%`
- **Query (expr)** — PromQL expression with variable placeholders

### Dynamic Vars (right panel)

Two modes, toggled:

- **Visual mode** (default): form to add/edit/remove variables. Each variable has: name, type (string, number, integer, boolean, enum), description, default value, required flag. Writes to `values.schema.json`.
- **Raw mode**: CodeMirror JSON editor showing `values.schema.json` directly.

### Template Files (tabs below fixed fields)

- CodeMirror YAML editor showing the generated Helm template
- Dropdown to switch between template files (e.g. `prometheus-rule.yaml`, `recording-rules.yaml`)
- Button to add new template file
- Template is auto-generated from fixed fields + dynamic vars, but editable

### Workflow

1. Template developer enters fixed fields (name pattern, description, expr)
2. Adds dynamic variables in the panel (db_name: string, warn: number, etc.)
3. UI auto-generates `templates/prometheus-rule.yaml` with `{{ range .Values.instances }}` loop
4. Developer can customize the generated template in CodeMirror
5. Save writes `Chart.yaml`, `values.yaml` (defaults), `values.schema.json`, and `templates/*.yaml`

## Alert User UX

### Layout

Left sidebar: chart selector, deployments list, template tree (unchanged from current).

### Main Area — Table

- UI reads `values.schema.json` for the selected chart/version
- Renders a table where:
  - **Columns** = variables from the schema (`db_name`, `tablespace_name`, `info`, `warn`, `critical`, `severe`)
  - **Column types** drive cell inputs: `string` -> text input, `number` -> number input, `boolean` -> checkbox, `string` + `enum` -> select dropdown
  - **Each row** = one instance (one entry in `values.yaml`'s `instances` array)
- Add row button appends a new instance with defaults from the schema
- Delete button per row

### Preview

On-demand "Preview" button calls `helm template` on the server with current values and displays the rendered PrometheusRule YAML in a modal.

### Save

Writes the user's `instances` array as the deployment's `values.yaml`. The deployment references the chart by name + version (same mechanism as current gitops-deploy).

## Server API Changes

### Template CRUD

The existing template API routes adapt to the new file structure:

- **GET /api/templates/:chart** — list chart names and versions
- **GET /api/templates/:chart/:version** — returns `{ chart: Chart.yaml, values: values.yaml, schema: values.schema.json, templates: { filename: content } }`
- **POST /api/templates/:chart/:version** — saves all files (chart, values, schema, template files)
- **DELETE /api/templates/:chart/:version** — removes version directory

### Schema Endpoint

- **GET /api/templates/:chart/:version/schema** — returns `values.schema.json` (used by alert user UI to generate form)

### Render

- **POST /api/render/:chart/:version** — takes values in body, runs `helm template`, returns rendered YAML

## Migration

Existing sample data under `sample/templates/alert-type/` and `sample/templates/alert-suite/` no longer maps to the new structure. New sample charts should be created that demonstrate the native Helm chart format. The old `alert-type`, `alert-type-pack`, and `alert-suite` template types are deprecated (editor pages already deleted).

## What This Eliminates

- Custom `vars` array format — replaced by `values.schema.json`
- `alertTypeName`, `alertTypeVersion`, `vars` fields in rule entries — no longer needed
- `packInstances` concept — already removed
- `alertSuite` wrapper object — values.yaml is just `instances: [...]`
- Custom Helm template generation logic — templates are standard Helm
- Duplicate `name`/`description` in values.yaml — comes from Chart.yaml
