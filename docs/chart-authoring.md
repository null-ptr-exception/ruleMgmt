# Chart authoring — optional variables

Charts consumed by AlertForge may be generated by the in-app template editor
or written by hand. Either way, the contract below governs how "a variable
was left empty" is represented, and what a template must do about it.

## The contract: empty = key absent

When a **non-required** variable (per the chart's `values.schema.json`
`required` arrays) is left empty in the UI, the key is **omitted entirely**
from the saved row in the deployment's values.yaml:

```yaml
cpu_alerts:
  - host: db-1
    warn_pct: 80        # set
  - host: db-2          # warn_pct left empty → key absent
```

Why key absence and not `null`:

- **JSON Schema**: `type` constraints only validate keys that are present.
  An absent key that is not in `required` passes `helm template` schema
  validation without any schema change.
- **Helm**: an explicit `null` in override values is a *delete-this-key*
  instruction during value coalescing (and its handling has varied across
  Helm versions). A `null` never reliably reaches templates. Do not
  hand-write `null` into values.yaml.

`required` in JSON Schema checks **key presence only** — it does not reject
empty strings. The UI currently backfills required variables with zero
values (`''` / `0` / `false`), so presence is always satisfied; stricter
enforcement in the UI is tracked as future work in #51.

## Guard convention for templates

Every reference to a non-required variable in a template must be guarded,
otherwise a missing key renders the literal string `<no value>` into your
rules. The canonical guard is `hasKey` — it is the exact inverse of the
save-side rule ("set" means "key present"):

```yaml
labels:
  severity: warning
  {{- if hasKey $row "team" }}
  team: "{{ $row.team }}"
  {{- end }}
```

(In templates without common vars the range element is `.`, so the guard is
`hasKey . "team"`.)

Plain truthiness (`{{ if $row.team }}`) is an acceptable shorthand **only**
for string variables where `""` can never be a meaningful value. It swallows
`0` and `false`, so never use it for numbers, booleans, or numeric enums —
a legitimate `warn_pct: 0` would silently disappear.

Required variables may be referenced unguarded — the key is always present.

Templates generated by the in-app editor follow this convention
automatically (`hasKey` guards on all non-required selectors).

## Common-vars fallback

Generated templates merge each row with the deployment's `_common` block:

```yaml
{{- $common := .Values._common | default dict }}
{{- range .Values.cpu_alerts }}
{{- $row := merge . $common }}
```

An omitted row key therefore inherits the `_common` value — *empty =
inherit common* is intended behavior, not an accident. Note that `merge`
(mergo semantics) also treats zero values (`0`, `""`, `false`) in the row as
empty slots to be filled from `_common`; if a zero value ever needs to
override a common var, the merge strategy and the guards must change
together (see #51 future work).

## Safety net

The Preview modal runs `promtool check rules` on the rendered output (#43).
An unguarded reference that leaks `<no value>` into a PromQL expression is
caught there before anything reaches the cluster — but labels and
annotations containing `<no value>` are syntactically valid, so promtool
will *not* catch a missing guard on a label. Guard your references.
